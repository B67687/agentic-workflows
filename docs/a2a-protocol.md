# A2A Protocol — Agent-to-Agent Communication Standard

## What It Is

The **Agent2Agent (A2A) Protocol** is an open standard (Linux Foundation, v1.0 March 2026, 23.7k★ on GitHub) that enables communication and collaboration between independent AI agents built using different frameworks, languages, or by different vendors.

It provides a common language for agents to:
- **Discover** each other's capabilities (via Agent Cards)
- **Negotiate** interaction modalities (text, files, structured data)
- **Manage** collaborative tasks with lifecycle tracking
- **Exchange** information securely without exposing internal state, memory, or tools

## Why It Matters to This Workspace

This workspace is an **AI Prompting Knowledge Base and Agent Orchestration Hub**. The A2A protocol is directly relevant because:

| Workspace Feature | A2A Concept | Connection |
|---|---|---|
| `agent-dispatch.sh` | A2A Client → Server | Our dispatch sends tasks to local agents (pi, codex, claude). A2A standardizes this for remote agents via HTTP. |
| Task lifecycle in `pipeline-run.sh` | Task lifecycle | A2A defines `SUBMITTED → WORKING → COMPLETED/FAILED/INPUT_REQUIRED`. Our pipeline tracks similar states. |
| `parley.sh` (multi-agent debate) | Multi-turn interaction | A2A specifies streaming and multi-turn for agent collaboration. Parley could use A2A as transport for agent panels. |
| `handoff.sh` | Task delegation | A2A formalizes how agents delegate tasks to each other. |
| Agent Card concept | Agent Card | A2A's AgentCard (JSON metadata) is similar to this workspace's `tools.sh` registry — declaring what an agent can do and how to reach it. |

## Key Concepts

### Core Actors

```
User → A2A Client (orchestrator) → A2A Server (remote agent)
```

- **User**: Human or automated service initiating a request
- **A2A Client**: Application/agent acting on the user's behalf
- **A2A Server**: Agent exposing an A2A-compliant HTTP endpoint

### Tasks (Stateful Work Units)

The fundamental unit of work. Each task has a unique ID and lifecycle:

```
SUBMITTED → WORKING → COMPLETED
                     → FAILED
                     → CANCELED
                     → REJECTED
                     → INPUT_REQUIRED (needs human input)
                     → AUTH_REQUIRED (needs authentication)
```

`INPUT_REQUIRED` and `AUTH_REQUIRED` are **interrupted** (not terminal) states — the task can resume after resolution. This is the A2P (agent-to-person) interaction point.

### Agent Cards (Discovery)

A JSON metadata document describing:
- Identity and provider
- Service endpoint URL
- Capabilities (streaming, push notifications, extended card)
- Authentication requirements
- Skills (named abilities with descriptions)

### Messages and Parts

- **Message**: A communication turn with role (`user` or `agent`) and one or more **Parts**
- **Part**: Content container — text, file (URL or inline bytes), or structured data (JSON)

### Artifacts

Tangible outputs generated by a task (documents, images, structured data). Distinct from messages — artifacts are the *deliverables*.

## Relationship to MCP

```
A2A (agent-to-agent)           MCP (agent-to-tool)
─────────────────────────────  ─────────────────────────────
Collaboration between agents   Connection to tools/data
Stateful, multi-turn           Stateless function calls
Opaque execution (black box)   Transparent tool interface
Task delegation and tracking   Tool invocation and results
```

**They are complementary, not competing.** An agent uses MCP internally to call tools, and A2A externally to collaborate with other agents. The A2A spec explicitly documents this relationship at `a2a-and-mcp/`.

From the A2A spec:
> "The distinction between A2A and MCP depends on what an agent interacts with. Tools and resources are MCP domain; agents are A2A domain."

## Interaction Patterns

| Pattern | How It Works | A2A Support |
|---|---|---|
| **Request/Response** | Client sends, server responds (polling for long tasks) | `sendMessage` + `getTask` |
| **Streaming** | Real-time updates via Server-Sent Events | `sendMessageStream`, `subscribeToTask` |
| **Push Notifications** | Server POSTs to client webhook on updates | Push notification config CRUD |
| **Multi-turn** | Agent pauses for input, continues on reply | `INPUT_REQUIRED` state |
| **File Exchange** | Upload/download parts via URL or inline bytes | Part `url` and `data` fields |

## How This Workspace Already Implements A2A-like Patterns

| A2A Feature | Workspace Equivalent | Gap |
|---|---|---|
| Agent Card | `scripts/tools.sh` (tool registry) | Tools.sh describes local scripts, not remote agent capabilities |
| Task lifecycle | `pipeline-run.sh` states | Pipeline uses simpler states, no task ID isolation |
| Async dispatch | `agent-dispatch.sh fire` | Works but only tested with pi-agent, no HTTP transport |
| Multi-turn | `parley.sh` debate panels | Parley uses OpenRouter, not A2A protocol |
| INPUT_REQUIRED | `handoff.sh`, `grill-me` skill | Human-in-loop patterns exist but aren't formalized as protocol states |
